# Component Types and the Type Function

Wardley's maps distinguish four types of component: **Activities**, **Practices**, **Data**, and **Knowledge**. The cheat sheet (Part 6) treats these as four named rows at the top of the characteristic table because each type *evolves* through the four stages with its own vocabulary ("Genesis → Custom → Product → Commodity" for an Activity, but "Novel → Emerging → Good → Best" for a Practice). In the map visualization, onlinewardleymaps renders each type with a different shape/colour.

The math model in Part 1 has one node type. This doc extends the tuple to include a type function and gives the type-dependent rules.

---

## 1. The four types

Summary from Simon Wardley's [*Wardley's Doctrine*](https://blog.gardeviance.org/2016/05/wardleys-doctrine.html) and his [Medium book](https://medium.com/wardleymaps):

| Type | What it is | Stage vocabulary (Genesis → Commodity) |
|---|---|---|
| **Activity** (A) | Something the organisation *does* — a capability, function, or service | Genesis · Custom · Product (+rental) · Commodity (+utility) |
| **Practice** (P) | *How* something is done — methodology, pattern, engineering culture | Novel · Emerging · Good · Best |
| **Data** (D) | Information used or produced | Unmodelled · Divergent · Convergent · Modelled |
| **Knowledge** (K) | Concepts, hypotheses, theories, accepted science | Concept · Hypothesis · Theory · Accepted |

Examples:

- *Activity*: "accept payment", "provide search", "compute"
- *Practice*: "agile software development", "continuous deployment", "chaos engineering"
- *Data*: "customer transaction records", "telemetry stream"
- *Knowledge*: "information theory", "the CAP theorem", "organisational psychology"

---

## 2. Extending the tuple

Add a type function $\tau$ to the Part 1 tuple:

$$\mathcal{M} = (V, E, U, \tau, \nu, \varepsilon, t)$$

Where:

$$\tau : V \setminus U \to \{A, P, D, K\}$$

Users ($U$) are not typed (they are the anchor, not a component). Every non-anchor component has exactly one type.

---

## 3. Type-dependent structural rules

### 3.1 Practices co-evolve with Activities

One of Wardley's climatic patterns: *"practices co-evolve with activities."* Mathematically, for every Practice $p$ there is typically at least one Activity $a$ whose dependency on $p$ exists and whose evolution is correlated with $p$'s. A soft constraint:

$$\text{for each } p \text{ with } \tau(p) = P: \quad \exists\, a \text{ with } \tau(a) = A, \, (a, p) \in E, \, \mathrm{corr}(\varepsilon_a(t), \varepsilon_p(t)) > 0$$

Concretely: "agile" emerges as a Practice because "software delivery" as an Activity demands it. Practices that don't support an Activity don't stick.

### 3.2 Knowledge underpins everything

Knowledge ($K$) typically sits below and supports Data, Practices, and Activities. Common pattern:

$$\text{Activity} \to \text{Practice} \to \text{Data} \to \text{Knowledge}$$

with each arrow a "depends on" edge. The visibility order $\nu(A) > \nu(P) > \nu(D) > \nu(K)$ is typical but not mandatory — a Practice built around user-visible interaction (e.g., customer research practices) can sit higher than the Activity it supports.

### 3.3 Type-respecting edges

Not every type-pair combination is meaningful. The common edge patterns:

| From → To | Meaning | Common? |
|---|---|---|
| A → A | An Activity depends on another Activity | Yes (normal value chain) |
| A → P | Delivering the Activity requires a Practice | Yes |
| A → D | Activity uses Data | Yes |
| A → K | Activity relies on Knowledge | Yes |
| P → P | A Practice builds on another | Yes |
| P → D / K | A Practice depends on Data / Knowledge | Yes |
| D → K | Data interpretation requires Knowledge | Yes |
| K → K | Knowledge builds on Knowledge | Yes |
| D → A | Data depends on Activity | Rare — usually means "data generated by" |
| K → A | Knowledge depends on Activity | Rare |

Edges from a lower type back up to a higher type (e.g., K → A) are unusual. A model can flag them for review but shouldn't forbid them.

## 4. Type-dependent evolution dynamics

The logistic form from Part 1 §5 applies to every type, but the **rate parameter** is typically type-dependent:

$$\frac{d\varepsilon_v}{dt} = r_{\tau(v)}(t) \cdot \varepsilon_v(t) \cdot (1 - \varepsilon_v(t))$$

Default rate ordering, from fastest to slowest:

$$r_A > r_P > r_D > r_K$$

Reasoning (informal):

- Activities evolve fastest because competition and supply/demand operate directly on them (they are what customers pay for).
- Practices evolve slower because they depend on accumulated professional experience and often on multiple successful Activity generations.
- Data evolves slower still because format and schema standardisation requires cross-organisational agreement.
- Knowledge evolves slowest — theories become "Accepted" on the timescale of careers or generations.

These defaults can be overridden per component when domain evidence warrants.

### 4.1 Type-specific inertia weights

The 17-form inertia drag (see the Inertia doc) can also be weighted by type:

$$c_v(t) = \sum_i \lambda_{i, \tau(v)} \cdot \iota_i(v, t)$$

Certain inertia forms hit certain types harder. For example:

- Sunk capital (inertia form #2) hits Activities hardest (you bought the hardware).
- Human capital (form #4) hits Practices hardest (your engineers trained in the old practice).
- Rewards and culture (form #16) hits Practices and Activities, not Data or Knowledge.

## 5. Type-dependent visibility seed

Part 1's visibility seed is purely distance-based. Type can refine it. A common refinement:

$$\nu_{\text{seed}}(v) = \frac{1}{1 + d(v)} \cdot \rho_{\tau(v)}$$

With default multipliers:

| Type | $\rho_\tau$ | Rationale |
|---|---|---|
| Activity | 1.0 | Users see Activities directly |
| Practice | 0.7 | Users see results of Practices but not usually the Practice itself |
| Data | 0.5 | Data is usually intermediate |
| Knowledge | 0.3 | Knowledge is background |

This keeps Activities toward the top of the map and Knowledge toward the bottom even when their graph distance from the user is similar. The mapper can still override per-component (Part 1 §3.5).

---

## 6. Type consistency as a map-quality check

Given a typed map, the following are warning signs:

1. **Practices with no dependent Activities** (violation of co-evolution). Check whether the Practice actually belongs on this map.
2. **Activities with no underlying Knowledge** beyond a few hops. May indicate the map hasn't been dug deep enough.
3. **Knowledge components at high visibility** ($\nu > 0.5$). Usually means the mapper has conflated "talked about" with "user-facing".
4. **Data components at Commodity (+utility) without a Knowledge dependency** at Accepted. Utility-grade data usually rests on well-accepted knowledge; if not, consider whether the data is really at that stage.

These are heuristics, not hard errors. Use them to prompt review, not to reject a map.

## 7. Worked example — a search feature

Component (type):

- `search results page` (A)
- `relevance ranking` (A)
- `user clickstream` (D)
- `BM25 / learning-to-rank` (K)
- `A/B testing discipline` (P)
- `data pipeline monitoring` (P)

Edges:
- (search results page, relevance ranking)
- (search results page, A/B testing discipline)
- (relevance ranking, user clickstream)
- (relevance ranking, BM25 / learning-to-rank)
- (user clickstream, data pipeline monitoring)

Type rules check:
- Two Practices (`A/B testing discipline`, `data pipeline monitoring`) each have an Activity or Data dependent — ✓ co-evolution satisfied.
- Knowledge (`BM25`) sits at the bottom — ✓ typical.
- No unusual edges (e.g., K → A pointing backwards).

Evolution rates: $r_A = 0.3$, $r_P = 0.15$, $r_D = 0.1$, $r_K = 0.05$. In a 5-year scenario, the Activities move several stage-widths, the Practices one or two, the Knowledge barely at all. This matches intuition: search results UI changes often; the underlying ML theory barely budges.

---

## 8. Backwards compatibility

Existing Part 1 / Part 2-5 / Part 6 maps can be read as $\tau \equiv A$ (all components are Activities). Nothing breaks. The type function is additive — maps that don't type their components behave exactly as before.
